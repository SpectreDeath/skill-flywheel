# Skill Flywheel Assessment Implementation Plan

This plan addresses all four recommendations from the assessment report.

## Goal

Implement the four key recommendations from the assessment report:
1. Fix telemetry dependency issues (opentelemetry version conflicts)
2. Modularize the monolithic unified_server.py into sub-routers
3. Offload evolution to background workers with real-time progress via WebSocket
4. Prepare for database scalability with SQLAlchemy

---

## Task 1: Fix Telemetry (opentelemetry version conflicts)

**Status**: Completed

### Context
The test suite is failing due to `ImportError: cannot import name 'ReadableLogRecord' from 'opentelemetry.sdk._logs'`. This is a version mismatch between `opentelemetry-sdk` and telemetry plugins.

### Implementation Steps
1. Add pinned `opentelemetry` dependencies to `requirements.txt`:
   ```
   opentelemetry-api==1.22.0
   opentelemetry-sdk==1.22.0
   opentelemetry-exporter-otlp==1.22.0
   opentelemetry-exporter-prometheus==1.22.0
   ```
2. Sync to `pyproject.toml` dependencies
3. Run tests to verify the fix

### Verification
Run the test suite to confirm no import errors.

---

## Task 2: Modularize unified_server.py

**Status**: Completed

### Context
`unified_server.py` is ~2800 lines and violates Single Responsibility Principle. It needs to be split into modular routers.

### Implementation Steps
1. Create router directory: `src/flywheel/server/routes/`
2. Create `src/flywheel/server/routes/__init__.py`
3. Create `src/flywheel/server/routes/adk.py` - ADK endpoints (L1/L2/L3 progressive disclosure)
4. Create `src/flywheel/server/routes/skills.py` - skill discovery and execution
5. Create `src/flywheel/server/routes/telemetry.py` - metrics and monitoring
6. Create `src/flywheel/server/routes/security.py` - security scanning (optional)
7. Refactor `unified_server.py` to import and include routers

### Verification
- Run `python -m pytest` to verify functionality
- Start server and test all endpoints

---

## Task 3: Offload Evolution to Background Workers

**Status**: Completed

### Context
Evolution runs in-process. Need to offload to Celery workers with WebSocket for real-time progress.

### Implementation Steps
1. Create Celery tasks in `src/flywheel/evolution/tasks.py`:
   - `run_evolution_generation` - execute one evolution step
   - `run_full_evolution` - execute full evolution loop
2. Modify `src/flywheel/evolution/evolver.py` to support async task submission
3. Add WebSocket endpoint in `src/flywheel/server/routes/evolution.py`:
   - `/evolution/ws` - WebSocket for real-time fitness progress
   - `/evolution/start` - POST to start evolution job
   - `/evolution/status/{job_id}` - GET job status
4. Update config in `src/flywheel/server/config.py` if needed

### Dependencies
- FastAPI WebSocket support (built into FastAPI)
- Redis already configured for Celery

### Verification
- Start Celery worker: `celery -A src.flywheel.evolution.tasks worker`
- Start server and test WebSocket connection
- Submit evolution job and verify progress streaming

---

## Task 4: Database Scalability (PostgreSQL with SQLAlchemy)

**Status**: Completed

### Context
SQLite is sufficient for metadata but will need migration as evolved skills population grows. Add SQLAlchemy support.

### Implementation Steps
1. Add SQLAlchemy to dependencies:
   ```
   sqlalchemy>=2.0.0
   asyncpg>=0.29.0  # PostgreSQL async driver
   ```
2. Create base models in `src/flywheel/db/models.py`:
   - `Skill` model
   - `SkillVersion` model
   - `EvolutionJob` model
3. Create database layer in `src/flywheel/db/repository.py`:
   - Abstract repository pattern
   - SQLite implementation (current)
   - PostgreSQL implementation (new)
4. Add database configuration in `src/flywheel/server/config.py`:
   ```python
   "database": {
       "type": "postgresql",  # or "sqlite"
       "connection_string": "..."
   }
   ```
5. Update skill manager to use repository pattern

### Verification
- Run migrations
- Verify skill CRUD operations work
- Test PostgreSQL connection (if available)
