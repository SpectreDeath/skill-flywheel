# SKILL: Recursive Logic Audit


## Implementation Notes
Content for ## Implementation Notes section to be added based on the specific skill requirements.

## Description

The Recursive Logic Audit skill is a self-correction mechanism that systematically evaluates an agent's reasoning process. By extracting step-by-step logic, detecting contradictions, and performing deep "Why?" probing, it prevents hallucinations and ensures that final outputs are grounded in verified premises.

## Purpose

Systematically audits the agent's own reasoning steps to detect logical fallacies, circular arguments, and unverified assumptions before generating final outputs.

## Capabilities

1. Step-by-step reasoning extraction.
2. Contradiction detection between internal premise and output.
3. Recursive "Why?" depth-probing (3-level minimum).

## Workflow

1. Log all intermediate reasoning steps.
2. Analyze links between steps for logical validity.
3. Generate a 'Confidence Log' highlighting potential weak points.

## Usage Examples

- Verifying a complex mathematical proof generated by the agent.
- Auditing a cross-file refactoring plan for circular dependency risks.
- Validating the chain of thought in a critical security assessment.

## Input Format

- **Chain of Thought**: The sequence of reasoning steps to be audited.
- **Audit Depth**: (Optional) Number of recursive levels to probe.

## Output Format

- **Confidence Log**: Detailed analysis of logical validity at each step.
- **Audit Result**: PASS/FAIL with specific fallacy markers if applicable.

## Configuration Options

- `audit_depth`: Minimum depth of reasoning to probe (default: 3).
- `fallacy_sensitivity`: Level of strictness in detecting logical flaws.

## Constraints

- MUST NOT proceed with final output if a "Critical Logical Break" is detected.
- MUST cite the specific step where the logic failed.

## Examples

### Example 1: Circular Argument Detection

**Input**: Reasoning chain that assumes 'A' is true because 'B' is true, and 'B' is true because 'A' is true.
**Output**: Audit Fail. Marker: "Circular Logic detected in steps 2 and 4."

## Error Handling

- If reasoning steps are too vague to audit, it requests clarification or a more detailed chain of thought.

## Performance Optimization

- Uses logical pruning to focus on the most controversial or high-impact reasoning branches.

## Integration Examples

- Typically run as a final "sanity check" before executing irreversible commands.

## Best Practices

- Always audit the reasoning before the implementation, not just the code itself.
- Use in tandem with "Cognitive Bias Guardrail" for complete quality assurance.

## Troubleshooting

- If legitimate but complex logic is being flagged, increase the `fallacy_sensitivity` threshold.

## Monitoring and Metrics

- Tracks "Hallucination Suppression Rate" and "Reasoning Depth Average".

## Dependencies

- Requires an internal formal logic validator.
